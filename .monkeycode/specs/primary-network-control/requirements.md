# 一网控制策略功能需求文档

## Introduction

一网控制策略模块是锅炉集中供热智慧管理系统的核心功能模块之一，旨在实现一次管网的智能调度与优化控制。该模块基于全局优化策略，综合考虑各换热站的热负荷需求和热源（锅炉）的供热能力，通过模糊控制算法实现热源的合理分配，最终达到全网供热均衡和节能的目标。系统需要与热源调控模块和换热站自控模块紧密集成，同时支持负荷预测和动态调度功能，为热力公司提供科学的调度决策依据。

## Glossary

- **一次管网**: 从热源（锅炉）到各换热站的热媒输送管道网络
- **热源**: 锅炉本体及附属设备，负责产生热能
- **换热站**: 一次管网与二次管网之间的热量交换设施
- **模糊控制**: 基于模糊逻辑的控制方法，适用于非线性、时变系统的控制
- **负荷预测**: 基于历史数据和算法预测未来热负荷需求
- **动态调度**: 根据实时工况调整热源分配策略
- **供热均衡**: 全网各换热站供热量与需求匹配，避免冷热不均
- **MQTT**: 轻量级物联网消息传输协议
- **时序数据库**: 专门存储时序数据的数据库，如InfluxDB

## Requirements

### Requirement 1: 热负荷需求采集

**User Story:** 作为热力公司调度人员，我希望系统能够实时采集各换热站的热负荷需求数据，为优化调度提供依据。

#### Acceptance Criteria

1. WHILE 系统处于运行状态, 系统 SHALL 每30秒从各换热站获取一次热负荷需求数据
2. WHEN 换热站上报负荷需求, 系统 SHALL 解析数据并更新全局负荷需求列表
3. IF 换热站数据采集中断超过60秒, 系统 SHALL 使用最近一次数据进行推算并发出告警
4. IF 连续3个采集周期未收到某换热站数据, 系统 SHALL 将该换热站标记为离线状态

### Requirement 2: 热源能力监测

**User Story:** 作为热力公司调度人员，我希望系统能够实时监测热源（锅炉）的供热能力，了解当前可用的热能资源。

#### Acceptance Criteria

1. WHILE 系统处于运行状态, 系统 SHALL 每30秒从热源调控模块获取一次锅炉运行状态数据
2. WHEN 锅炉运行状态发生变化, 系统 SHALL 实时更新热源能力模型
3. IF 锅炉发生故障或停机, 系统 SHALL 立即重新计算剩余热源能力并调整调度策略
4. IF 热源能力无法满足最低负荷需求, 系统 SHALL 发出告警提示调度人员采取措施

### Requirement 3: 模糊控制算法实现

**User Story:** 作为热力公司技术人员，我希望系统能够采用模糊控制算法进行热源分配优化，适应一次管网非线性和时变特性。

#### Acceptance Criteria

1. WHEN 系统执行调度计算, 系统 SHALL 使用模糊控制算法处理输入变量（负荷需求偏差、热源能力偏差、管网压力偏差）
2. WHILE 模糊控制系统处于运行状态, 系统 SHALL 根据模糊规则库输出调节指令
3. IF 模糊控制器检测到工况剧烈变化, 系统 SHALL 自动调整量化因子和比例因子
4. IF 模糊控制输出超出安全范围, 系统 SHALL 自动限幅并告警

### Requirement 4: 热源分配优化

**User Story:** 作为热力公司调度人员，我希望系统能够根据各换热站需求和热源能力，自动计算最优的热源分配方案。

#### Acceptance Criteria

1. WHEN 执行热源分配优化, 系统 SHALL 综合考虑换热站负荷需求、热源产能、管网输送能力等因素
2. WHILE 优化算法处于运行状态, 系统 SHALL 每5分钟执行一次优化计算
3. IF 优化计算结果收敛, 系统 SHALL 生成热源分配指令并下发到热源调控模块
4. IF 优化计算结果不收敛, 系统 SHALL 保持当前分配方案并告警提示人工干预

### Requirement 5: 负荷预测

**User Story:** 作为热力公司调度人员，我希望系统能够预测未来热负荷需求，提前调整热源分配策略。

#### Acceptance Criteria

1. WHEN 系统执行负荷预测, 系统 SHALL 基于历史数据使用时间序列预测算法预测未来2小时负荷
2. WHILE 负荷预测系统处于运行状态, 系统 SHALL 每小时更新一次预测数据
3. IF 预测误差超过实际值的20%, 系统 SHALL 自动调整预测模型参数
4. IF 预测到负荷将大幅上升, 系统 SHALL 提前发出预警提示调度人员

### Requirement 6: 动态调度

**User Story:** 作为热力公司调度人员，我希望系统能够根据实时工况动态调整调度策略，适应负荷变化。

#### Acceptance Criteria

1. WHEN 检测到负荷变化超过10%, 系统 SHALL 触发动态调度重新计算
2. WHILE 动态调度处于自动模式, 系统 SHALL 根据实时数据持续调整控制参数
3. IF 负荷变化剧烈（超过30%）, 系统 SHALL 启动快速调节模式，缩短控制周期
4. IF 动态调度连续3次不收敛, 系统 SHALL 切换到安全模式并告警

### Requirement 7: 一网流向可视化展示

**User Story:** 作为热力公司调度人员，我希望通过可视化界面直观查看一次管网的流向和热源分布情况。

#### Acceptance Criteria

1. WHEN 用户打开一网监控页面, 系统 SHALL 展示一次管网拓扑图和热源分布
2. WHILE 页面处于打开状态, 系统 SHALL 每10秒刷新一次管网流向和温度数据
3. WHEN 用户点击某个管段, 系统 SHALL 显示该管段的详细信息（流量、温度、压力）
4. IF 管段温度异常, 系统 SHALL 在可视化界面上以红色标记该管段

### Requirement 8: 热源分布可视化展示

**User Story:** 作为热力公司调度人员，我希望通过可视化界面查看各热源的运行状态和供热占比。

#### Acceptance Criteria

1. WHEN 用户打开热源分布视图, 系统 SHALL 展示各锅炉的运行状态和出力比例
2. WHILE 视图处于打开状态, 系统 SHALL 每10秒刷新一次热源数据
3. WHEN 用户点击某个锅炉, 系统 SHALL 显示该锅炉的详细运行参数
4. IF 锅炉运行参数超出正常范围, 系统 SHALL 在界面上以相应颜色标记

### Requirement 9: 与热源调控模块集成

**User Story:** 作为热力公司调度人员，我希望一网控制模块能够与热源调控模块协同工作，实现整体优化。

#### Acceptance Criteria

1. WHEN 一网控制模块生成热源分配指令, 系统 SHALL 通过API下发到热源调控模块
2. WHILE 两系统通信正常, 系统 SHALL 实时同步运行数据和调度指令
3. IF 热源调控模块通信中断超过60秒, 系统 SHALL 自动切换到本地应急模式
4. IF 热源调控模块返回执行失败, 系统 SHALL 记录失败原因并重新计算优化方案

### Requirement 10: 与换热站自控模块集成

**User Story:** 作为热力公司调度人员，我希望一网控制模块能够向换热站下发调节指令，实现全网协调控制。

#### Acceptance Criteria

1. WHEN 一网控制模块生成换热站调节指令, 系统 SHALL 通过API下发到对应换热站
2. WHILE 换热站处于远程调度模式, 系统 SHALL 优先执行一网控制模块的指令
3. IF 换热站返回执行异常, 系统 SHALL 记录异常信息并调整调度策略
4. IF 换热站本地控制优先级更高, 系统 SHALL 暂停对该换热站的远程控制

### Requirement 11: 调度策略配置

**User Story:** 作为热力公司技术人员，我希望系统能够灵活配置调度策略参数，适应不同工况需求。

#### Acceptance Criteria

1. WHEN 用户修改调度策略参数, 系统 SHALL 立即更新参数并应用到控制算法
2. IF 用户修改模糊控制规则, 系统 SHALL 验证规则一致性后再生效
3. IF 用户切换调度模式, 系统 SHALL 记录切换日志并执行相应的控制逻辑
4. IF 参数配置超出安全范围, 系统 SHALL 拒绝修改并提示正确范围

### Requirement 12: 历史数据存储与查询

**User Story:** 作为热力公司调度人员，我希望系统能够存储历史调度数据，便于分析和优化。

#### Acceptance Criteria

1. WHILE 系统处于运行状态, 系统 SHALL 每5分钟存储一次调度结果数据
2. WHEN 用户查询历史调度数据, 系统 SHALL 支持按时间范围、换热站等条件筛选
3. WHEN 用户导出历史数据, 系统 SHALL 支持CSV和Excel格式导出
4. IF 历史数据存储失败, 系统 SHALL 记录失败日志并重试存储

### Requirement 13: 调度日志记录

**User Story:** 作为系统运维人员，我希望系统能够记录完整的调度日志，便于故障排查和审计。

#### Acceptance Criteria

1. WHILE 系统处于运行状态, 系统 SHALL 记录所有调度操作的详细日志
2. WHEN 调度策略发生变化, 系统 SHALL 记录策略变更前后对比数据
3. IF 发生告警或异常, 系统 SHALL 记录完整的上下文信息用于分析
4. IF 日志存储空间不足, 系统 SHALL 自动清理超过90天的历史日志

### Requirement 14: 调度效果评估

**User Story:** 作为热力公司管理人员，我希望系统能够评估调度效果，为优化调度策略提供依据。

#### Acceptance Criteria

1. WHEN 用户请求调度效果评估, 系统 SHALL 计算全网供热均衡度和能耗指标
2. IF 评估结果显示供热不均衡度超过10%, 系统 SHALL 建议调整调度策略
3. IF 能耗指标优于历史同期, 系统 SHALL 生成优化经验报告
4. IF 调度效果评估异常, 系统 SHALL 提示数据不足或计算错误

## Non-Functional Requirements

### Performance Requirements

- 系统响应时间：界面操作响应时间不超过500毫秒
- 优化计算周期：5分钟执行一次全局优化
- 动态调度响应：负荷变化触发后1分钟内完成重新计算
- 并发用户：支持至少30个用户同时访问调度界面

### Reliability Requirements

- 系统可用性：达到99.5%
- 数据采集成功率：不低于99%
- 调度指令执行成功率：不低于98%
- 优化计算收敛率：不低于95%

### Security Requirements

- 用户认证：支持用户名密码认证
- 权限控制：支持角色-based权限管理，普通用户只读，管理员可操作
- 操作审计：记录所有控制操作日志
- 数据传输：支持HTTPS加密传输

### Integration Requirements

- 与热源调控模块集成：通过REST API进行数据交互
- 与换热站自控模块集成：通过REST API进行指令下发和状态同步
- 数据格式：JSON格式进行数据交换
- 通信协议：HTTP/HTTPS协议
